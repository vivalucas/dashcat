# 规划决策记录

这里记录已经做过的重要规划和取舍，便于后续继续沿用。

## ADR-001：纯 AppKit，不引入 SwiftUI

- **状态**：已采用
- **背景**：菜单栏和浮窗功能简单，但要求低占用
- **方案**：用 AppKit 直接实现状态项、菜单和面板
- **备选方案**：SwiftUI + AppKit
- **依据**：更轻、更直接，符合 DashCat 的定位

## ADR-002：剪贴板使用 SQLite

- **状态**：已采用
- **背景**：需要保存历史、做过期清理、按需搜索
- **方案**：SQLite + 图片文件夹
- **备选方案**：JSON / Core Data / SwiftData
- **依据**：清理和查询都更省资源

## ADR-003：搜索采用包含匹配

- **状态**：已采用
- **背景**：用户希望输入关键词就能找到相关条目
- **方案**：大小写不敏感的普通包含匹配
- **备选方案**：全等匹配、模糊搜索、正则
- **依据**：简单、好用、实现成本低

## ADR-004：电量状态项独立显示

- **状态**：已采用
- **背景**：刘海屏和窄菜单栏空间紧张
- **方案**：独立 `NSStatusItem`，只显示极窄数字
- **备选方案**：复用猫咪状态项或显示完整百分号
- **依据**：不挤占主状态项空间

## ADR-005：Finder 新建文件只走 DashCat 菜单

- **状态**：已采用
- **背景**：需要低成本创建当前文件夹内的新文件
- **方案**：AppleScript 读取 Finder 当前目录，DashCat 菜单触发
- **备选方案**：Automator Quick Action、Finder Sync Extension
- **依据**：前者不满足空白处右键场景，后者维护成本高

## ADR-006：不做自建 API

- **状态**：已采用
- **背景**：项目是本地菜单栏工具
- **方案**：只使用系统框架和网页入口
- **备选方案**：HTTP 服务 / 本地接口层
- **依据**：没有必要增加额外边界

## 2026-06-08：综合评审修复与用户体验优化计划

- **状态**：已确认，进入实现
- **背景**：综合评审发现防休眠失败回退、剪贴板图片清理一致性、文档版本口径等问题，同时确认若干剪贴板和 Finder 流程优化。
- **方案**：
  1. 修复防休眠 assertion 创建失败后 UI / 持久化状态仍显示开启的问题，失败时回退关闭。
  2. 永久保留剪贴板历史时仍执行孤儿图片清理，避免长期残留文件。
  3. 调整图片条目删除顺序，先确认数据库记录删除成功，再删除图片文件和缩略图。
  4. 清除历史增加确认弹窗，提供“清除未固定项”和“清除全部”两个明确选项。
  5. 剪贴板面板增加空状态提示，区分无历史和搜索无结果。
  6. 图片缩略图改为异步加载，减少打开面板和滚动时的主线程磁盘读取。
  7. Finder 新建文件弹窗允许输入文件名，保留重名自动递增。
  8. 更新检查期间禁用菜单项并显示进行中状态，避免重复请求。
  9. README 按当前软件实际颜色和流程同步。
- **依据**：这些改动都保持纯 AppKit、菜单即配置和低依赖路线，不引入新架构。
